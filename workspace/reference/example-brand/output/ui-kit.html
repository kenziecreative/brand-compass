02

Buttons

Buttons that mean business. Primary actions use solid fills in navy or teal. Secondary buttons hold the border without demanding attention. Ghost buttons step back entirely. The coral destructive variant only appears when something is about to be permanently removed.

Primary — Filled

Secondary — Outlined

Ghost — Text Only

Sizes

Hover States (simulated)

Disabled

Destructive — Coral

Pattern Rules

  • Primary Navy: The default action button. Use for primary CTAs that carry weight — "Start Audit," "Submit Report," "Save Changes."
  • Primary Teal: Forward-motion actions. Use when the action moves the user ahead — "View Report," "Continue," "Next Step."
  • Secondary: Supporting actions that sit alongside a primary. "Export," "Schedule," "Duplicate." Never use alone as the only CTA.
  • Ghost: Low-emphasis actions — cancel, dismiss, learn more. Use in toolbars, card footers, and inline contexts.
  • Destructive: Permanent removal only. Delete, revoke, remove. Never use coral for warnings — that is the alert system's job.
03

Form Elements

Compliance workflows run on forms. Every input uses a warm-gray background to distinguish it from the page, a navy focus ring for clarity, and coral for errors that demand attention. The pattern is always: label, input, helper text.

Legal entity name as registered.
Primary contact for this audit engagement.
Please enter a valid 9-digit EIN (XX-XXXXXXX).
Domain verified successfully.
Auto-generated. Cannot be changed.

Pattern Rules

  • Layout: Use two-column layouts for related form groups. Single-column for sequential flows (wizards, onboarding).
  • Labels: Always above the input, never inline. Labels use Inter 600 at 14px.
  • Helper text: Below the input in slate. Switches to coral on error, teal on success.
  • Error messages: Appear immediately on validation. Always explain what is wrong and how to fix it — "Enter a valid 9-digit EIN" not just "Invalid input."
  • Disabled inputs: Use for system-generated values only. If a field should not be edited, disable it and explain why in helper text.

Pattern Rules

  • Simple cards: Use for service listings, feature descriptions, and content grouping where no action is needed.
  • Image cards: Use when a visual preview adds context — dashboard screenshots, report previews, integration logos.
  • Footer cards: Use when the card needs one or two clear actions. Primary action on the left, secondary on the right.
  • Stat cards: Use for key metrics and KPIs. Numbers use DM Serif Display for visual weight. Labels are uppercase Inter.
05

Badges & Tags

Small labels that carry precise meaning. In a compliance interface, a badge is not decoration — it tells you whether something passed, failed, or needs attention. Every color maps to a specific semantic.

Filled Badges

Audit Ready Compliant Archived Non-Compliant

Outline Badges

SOC 2 PCI DSS ISO 27001 HIPAA

Status Tags

Active Pending Review Needs Attention Complete

Pattern Rules

  • Badges label categories or static attributes — framework type, role, tier. They do not change during a session.
  • Tags indicate live status with a colored dot. They change as the underlying data changes — active, pending, alert, complete.
  • Filled badges for high-density interfaces (tables, lists) where the badge needs to stand out from surrounding text.
  • Outline badges for lower-density contexts (card headers, metadata lines) where a subtle label is sufficient.
  • Color mapping: Teal = passing/active. Coral = failing/attention. Navy = complete/authoritative. Slate = neutral/archived.

Pattern Rules

  • Density: Use borders between items in dense lists (10+ items). Use spacing alone for shorter lists (3–5 items) where visual breathing room matters more.
  • Metadata placement: Right-aligned for dates, counts, and single values. Inline badges for status.
  • Action buttons: Keep to 1–2 actions per item. Primary on the left, secondary on the right. Use small size to avoid visual competition with the content.
  • Hover state: Subtle warm-gray background shift on hover for clickable items. No hover for read-only lists.
